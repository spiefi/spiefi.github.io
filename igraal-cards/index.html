The Core Argument

Cross-team usage = system-level component

The designer comments (from Iza) flagged cards where multiple teams use the same pattern. That's not a coincidence — it's evidence. When CO and CCE both build a hero banner independently, you don't have two different components. You have one component built twice, with drift. The fix is one shared component in Nalu.

Patterns That Aren't Cards

Three patterns appear in the PDF but belong outside the card system. Here's what they look like and why they're excluded.

Filter / Facet Panel
Not numbered in PDF
Typical filter sidebar — iGraal search/category pages
Filtrer par catégorie
Why not a card
This is an input control — checkboxes, toggles, apply/reset. Cards are display containers that present content. If you call this a card, card components inherit form semantics (role=checkbox, form state, validation). That's a category error. It belongs in a Filter component, not the card system.
Checkout / Balance Summary
Not numbered in PDF
Appears on: account / cashback balance pages
Mon solde cashback
Cashback ENGIE€40.00
Cashback AliExpress€2.50
Bonus parrainage+€5.00
Total disponible€47.50
Why not a card
Cards are reusable units — you drop 12 into a grid. This appears once, with hardcoded business logic baked in (line items, running total, withdrawal CTA). It's a page section tied to account logic. Putting it in Nalu would couple the design system to a specific financial flow.
Empty State
Already in Nalu ✅
Appears on: empty search results, no offers state
🔍
Aucun résultat trouvé
Essayez de modifier vos filtres ou votre recherche
Why not a card — already solved
Nalu already ships EmptyState — illustration + heading + body + CTA. Building it again as a "card" means two components doing the same job. The whole point of this audit is reducing duplication, not creating more.
