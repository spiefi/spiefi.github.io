JIT Component Palette — Design Spec v1.0

Smart Discovery Components

10 composable building blocks that the JIT engine assembles into answer screens. Each component is self-contained, responsive, and designed for VStack assembly.

Design System Rules

Component Architecture

Every JIT answer screen is a VStack of these components. The Intent Engine scores each for relevance (0–1) and assembles those above 0.6 threshold. Order: relevance × user preference × recency.

Layout
VStack only. Full-width cards. 16px horizontal padding. 12px vertical gap between components.
Corners
14px border radius on cards. 10px on inner elements. 100px on pills/tags.
Typography
Inter. Rates: 18px/900. Names: 14px/700. Body: 13px/400. Labels: 11px/600 uppercase.
States
Every component has: Default, Loading (skeleton), Error (graceful fallback), Empty (never blank).
Color System
Green = cashback/positive. Amber = urgency. Rose = alert/expired. Accent = interactive. Cyan = data.
Accessibility
Semantic HTML. ARIA labels on all interactive elements. Logical tab order. Min touch target: 44×44px.

💰 Cashback Merchant Card

Primary building block. Shows ranked merchants for a category/query with cashback rates, active codes count, and one-tap activation. Always appears first in results.

Data Requirements
merchant rate category activeCodeCount logoUrl deeplink rateHistory[] isBookmarked
Variants
Compact: Single merchant row (in Browse feed cards)
List: 2–5 merchants ranked (default in Search/Ask results)
Highlighted: Top merchant with expanded detail + "Best Deal" badge
Interactions
• Tap merchant row → expand to show codes + gift card option
• Tap CTA → activate cashback + redirect to merchant
• Long press → save/bookmark merchant
• Rate badge shows trend indicator (↑↓→) from rateHistory
States
Loading
Skeleton: 3 rows with pulsing placeholders
Empty
Never empty — fallback to trending merchants
Error
Show cached data + "rates may be outdated"
Expired
Grey out + "Rate changed" + show current rate
Design Rationale
Rate transparency in results (steal from TopCashback). Ranked by cashback % by default — user can re-sort. Active code count creates a secondary incentive to tap.

🎟️ Promo Code Card (Transparent)

Shows tested promo codes with success rates and full transparency. Directly counters the Honey scandal — users see how many codes were tested, not just the ones that work.

Variants

📊 Cross-Retailer Comparison

Side-by-side comparison of 2–3 merchants for the same query. Shows cashback + codes + estimated total savings. Answers the question: "Where should I actually buy this?" Steal from Capital One Shopping.

Data Requirements
merchants[] rates[] bestCodes[] estimatedPrice totalSavings[] winner
Variants
2-column: Direct A vs B (most common)
3-column: Up to 3 merchants (max — more gets cluttered on mobile)
Summary: Just the winner badge with delta ("Zalando saves €22 more")
Interactions
• Amount input field: user can type their expected purchase amount for live calculation
• Tap any column → go to that merchant's Merchant Card
• Winner row highlighted with green background
When to Show
Relevance score > 0.6 when query matches 2+ merchants in same category. Always in Ask mode for product queries. In Search mode when multiple results exist. Never for single-merchant results.

For Savvy Maximizers. Breaks down every savings layer: cashback + promo code + gift card bonus = total. Interactive amount input. Makes iGraal's unique stacking capability visible and compelling.

🎁 24h Gift Card Card

iGraal's killer differentiator. Rakuten = 120 day wait. iGraal = 24 hours via gift cards. This component must appear in every result set where gift cards are available. First-class, never hidden.

Data Requirements
merchant giftCardRate denominations[] deliveryTime available standardWaitDays
Visual Treatment
Green-tinted card with border. Must visually stand out from standard merchant cards. The green gradient + ⏱️ "24h" creates instant recognition. Always shows the comparison: "24h vs. 30-90 day wait."
Variants
Full: As shown — standalone with amount calculator
Inline: Single line in Calculator or Merchant Card: "⚡ Get cashback in 24h via gift card"
Nudge: Small banner at bottom of any result: "Want faster cashback?"
When to Show
Always when the merchant offers gift cards. This is the trust differentiator — the one feature that directly counters the #1 category frustration (payout waits). Relevance score override: minimum 0.7 when available.

📈 Rate Trend Graph

Shows cashback rate history over 12 months. Empowers Savvy Maximizers to time purchases. "Is now a good time to buy?" answered visually. Creates a reason to check back.

Data Requirements
merchant rateHistory[{date, rate}] currentRate trend percentile insight
Insights (Auto-Generated)
• "12-month high" / "At average" / "Below average — usually higher in [month]"
• "Rate increased 3x in the last week" (spike alert)
• "Usually drops after Black Friday" (seasonal pattern)
When to Show
In Ask mode for "should I buy now?" or "best time" queries. In Search/Browse when merchant has volatile rate history. Always in expanded Merchant Card detail view. Never for merchants with flat/stable rates (adds no value).

✅ Tracking Confirmation

Post-purchase confirmation that cashback was tracked. The #1 requested feature across the entire cashback category. Eliminates the anxiety gap between buying and seeing cashback appear. Critical trust builder.

Data Requirements
orderId merchant amount rate earnedAmount status estimatedPayoutDate giftCardAlternative
Timing
Must appear within seconds of purchase completion. Not minutes, not hours — seconds. This is the single most impactful trust signal. If tracking is delayed, show "Verifying your cashback..." with spinner, then confirm.
States
Confirmed ✅
Green — tracked successfully
Verifying ⏳
Amber — checking with merchant
Issue ⚠️
Rose — "didn't track" + one-tap claim
Available 💰
Green pulse — ready to withdraw
Key Design Decision
The "Issue" state must include a one-tap manual claim — not a support ticket. Research shows the current category pattern (file a claim → wait days → maybe get credit) is the #1 churn driver. Instant claim = instant trust.

🏆 Quest Card

Adapted from ShopBack's quest system. Time-limited challenges that drive repeat engagement. Critical rule: quests must be based on user's actual shopping patterns, not merchant payments. Gamification that feels fair.

Data Requirements
questTitle description reward progress total completedMerchants[] eligibleMerchants[] expiry category
Quest Design Rules
Never: "Spend €200 at [specific merchant]" — feels like an ad
Always: "Shop at 3 fashion brands" — user chooses which brands

• Achievable within the time window (don't set impossible goals)
• Reward proportional to effort (€5 for 3 shops, not €1)
• Based on categories user already shops in (not random)
• Max 2 active quests at a time (avoid overwhelm)
States
Active
Progress bar + remaining steps + countdown
Complete 🎉
Celebration animation + reward credited
Expiring
Amber highlight + "1 day left!"
Failed
"Time's up. New quest available tomorrow."

🔔 Smart Alert Card

Turns zero-result into retention. When no exact match exists, offer to alert the user when one appears — plus show trending alternatives. Never show an empty screen. Every dead end becomes a hook.

Data Requirements
query suggestedAlerts[] channel trendingAlternatives[] relatedCategories[]
Alert Options (Chip-Based)
Auto-generated from the failed query:
• Brand-specific: "Nike cashback > 8%"
• Category-wide: "Any sneaker deal"
• Price-bound: "Under €80"
User taps chips to customise. One tap to set. Push notification when triggered.
Always Include
"Meanwhile, trending now" section below the alert — ensures the screen is never empty. Shows 2-3 popular deals from related categories. The user came for a deal; give them something even if it's not exactly what they asked for.
When to Show
When Intent Engine returns < 0.4 relevance for all components. When merchant doesn't exist in iGraal's network. When all deals for a query are expired. Target: this component should be seen rarely — Ibotta benchmark of −73% zero results means most queries should have real results.
